# Section 021: Blueprint Visualizer

> **A self-contained HTML artifact that renders a DAP blueprint as an interactive visual reference** — an install-flow execution surface, static risk analysis, plugins and inputs panels, and a source viewer, all in a single file that opens in any browser with no server or internet connection required.

---

## What Is the Blueprint Visualizer?

The Blueprint Visualizer is a single-page HTML file generated by Blueprint Assist that gives you a rich, interactive view of any DAP blueprint. It embeds all blueprint data (nodes, relationships, inputs, plugins, install flow, risk findings, and raw source files) directly into the HTML, so the file is fully self-contained and portable — no server, no CDN, and no DAP access required to open it.

Once generated, share it over email, attach it to a PR, open it in a demo, or embed it in documentation.

> **New in v0.28.2 — the visualizer is now a self-contained React app.** `dap-bpa blueprint visualize` generates the React-based diagram (install-flow steps, platform detection, plugins panel, and a dedicated Risk Analysis panel) instead of the legacy Cytoscape graph. React is inlined into the output HTML, so there are **no CDN requests at runtime** and the file works fully offline.

---

## How to Run It

**Manual command:**

```bash
dap-bpa blueprint visualize --file ./my-blueprint          # a directory
dap-bpa blueprint visualize --file ./blueprint.yaml        # a single file
dap-bpa blueprint visualize --file ./my-blueprint --output ./custom-name.html
```

Output defaults to `<blueprint-name>-visualizer.html` (derived from the file or directory name) in the current working directory, or the path given by `--output`.

> The `--file` path must be **within the current working directory** — `dap-bpa` rejects paths that resolve outside it.

**Automatic generation:**

It runs automatically after a successful upload:

```bash
dap-bpa orchestrator blueprints upload --file ./my-blueprint --id my-bp --revision 1.0.0
# ✓ Blueprint visualization generated: my-blueprint-visualizer.html
```

To disable auto-generation, set the environment variable `BPA_DISABLE_VISUALIZATION=true`. If visualization fails during an upload, the upload itself still succeeds — you can regenerate manually later.

**Common uses:**

- Preview the topology and review risks *before* uploading.
- Sanity-check node/relationship wiring after authoring or editing a blueprint.
- Feed the risk findings into remediation — the related `blueprint-risk-fix` workflow applies fixes from the visualizer / risk report.

## Quick Reference: What It Generates

A self-contained HTML file (title: **Blueprint Architecture Visualizer**) with:

| Area | Description |
| --- | ----------- |
| **Blueprint Execution Surface** | The central canvas — nodes and relationships laid out by install order, animatable through the install/uninstall flow |
| **Infrastructure Map** | A platform-detected view selector that groups nodes by the target platform's resource hierarchy |
| **Step Details** | The ordered install-flow steps, with parallel-step indicators for nodes that deploy together |
| **Risk Analysis** | Static security, reliability, operability, and lifecycle findings with severity, category, rule reference, confidence, and remediation guidance |
| **Plugins** | Every imported plugin with its version constraint |
| **Input Groups** | All blueprint inputs organized by group |
| **Sub-Deployment** | For composer blueprints, the resolved `ServiceComponent` sub-flow (referenced blueprint's nodes in install order) |
| **Source** | Embedded YAML with syntax highlighting |

See [How to Run It](#how-to-run-it) above for the manual command, default output naming, and automatic generation on upload.

## The Visualizer at a Glance

The app is a single unified view rather than a set of tabs. The centre of the screen is the **Blueprint Execution Surface** — a graph of node templates connected by their relationships (`depends_on`, `contained_in`, `connected_to`). Surrounding it are collapsible panels that describe the blueprint from different angles.

### Blueprint Execution Surface (centre)

- Nodes are rendered as coloured cards, positioned by their place in the install order and coloured by category (compute, storage, automation, network, security, snapshot, service component, generic).
- Relationship lines connect dependent nodes; external resources (`use_external_resource: true`) are marked distinctly.
- Play through the **Install** flow (`▶ Install`) or reverse it (`◀ Uninstall`) to watch nodes activate step by step — useful for training and demos because it makes the TOSCA execution model concrete.
- Nodes that share an install step are **badged as parallel** so you can see at a glance where the platform deploys resources concurrently versus sequentially.
- Risk-bearing nodes carry a severity badge; selecting a finding in the Risk Analysis panel focuses the affected node on the surface.

### Infrastructure Map (platform detection)

BPA detects the target platform from the blueprint's node types and labels and offers an **Infrastructure Map** view that groups nodes into that platform's natural resource hierarchy. The surface header shows the detected `scopeLabel` (for example, *Dell Automation Platform Environment*).

### Step Details

An ordered list of the derived install-flow steps. Each step lists the nodes that run in it and whether they run **in parallel** (grouped together) or **sequentially** (one step after another), mirroring the animation on the execution surface.

### Node palette

Each node type is described with a short summary and a **"Best for:"** note (for example, *Looks up a pre-existing …* or *Runs an automation task …*) so reviewers can understand the role each node plays without reading the YAML.

### Source

A syntax-highlighted YAML viewer for every source file in the blueprint — the main entrypoint plus any imported sub-files (infrastructure, application, inputs, outputs). File tabs across the top switch between files.

## Risk Analysis Panel

New in v0.28.2, the **Risk Analysis** panel runs a static risk engine over the parsed blueprint at generation time and renders the findings in a dedicated panel. The engine evaluates roughly two dozen rules across four categories — **security**, **reliability**, **operability**, and **lifecycle** — covering things like hardcoded credentials, isolated nodes, missing delete lifecycle, resource naming collisions, unpinned plugins, and input hygiene.

The panel has three parts:

- **Severity summary chips** — a count of findings by severity: **High**, **Medium**, **Low**, and **Info**.
- **Filter chips** — filter the cards by severity or by category (security, reliability, operability, lifecycle) to focus on what matters. When a filter hides everything, the panel shows *"No risks matched the current filter."*
- **Risk cards** — one card per finding. Each card shows:
  - A **severity badge** (High / Medium / Low / Info)
  - A **category tag** (security / reliability / operability / lifecycle)
  - A **rule reference** (e.g. `SC-001`) that links to per-rule documentation in `docs/RISK_RULES.md`
  - A **confidence** indicator (high / medium / low)
  - A **fixable** badge when an auto-fix is available
  - Title, description, and an actionable **recommendation**
  - A **node link** (for node-scoped findings) that focuses the affected node on the execution surface
  - A collapsible **Show Fix** snippet with a **Copy** button, when a suggested YAML fix exists

Every finding links back to `docs/RISK_RULES.md` (in the [dap-bpa toolkit repo](https://github.com/fusion-e/ai-bp-toolkit/blob/main/docs/RISK_RULES.md)) via a per-rule anchor (e.g. `RISK_RULES.md#sc-001`), so a reviewer can read the full rationale for any rule.

This panel makes the visualizer useful for **blueprint review and pre-deployment sanity checks** — reviewers can see potential security, reliability, operability, and lifecycle issues without opening the YAML.

### Risk Rules

The risk engine ships a static rule catalog. The rules evaluated for a standard (resource-mode) blueprint include:

| Rule | Category | Severity | Detects |
| ---- | -------- | -------- | ------- |
| `SC-001` | Security | High | Hardcoded credentials in node properties (recommends `get_secret`) |
| `RC-001` | Reliability | High | Resource naming collisions — multiple nodes managing the same resource |
| `ND-001` | Reliability | Medium | Isolated node with no dependency relationships |
| `ND-002` | Operability | Medium | External-resource dependency that must exist before deployment |
| `ND-003` | Lifecycle | Medium | Orphaned attachment risk — attachment/mapping that may leak on failed uninstall |
| `ND-004` | Lifecycle | Medium | Missing delete lifecycle — `create` without a matching `delete` |
| `ND-005` | Reliability | Medium | External source capacity impact |
| `ND-006` | Reliability | Medium | Oversized blast radius — a large number of managed resources |
| `TD-001` | Lifecycle | Medium | Outdated / unsupported `tosca_definitions_version` |
| `TD-002` | Operability | Medium | Legacy `nativeedge.*` references |
| `IM-003` | Operability | Low | Unpinned plugin version (no version constraint) |
| `IG-001` | Operability | Low | Inputs declared but no `input_groups` |
| `IN-003` | Operability | Info | Input missing a `display_label` |
| `IN-004` | Operability | Info | Input missing a description |
| `IN-005` | Operability | Info | Input missing an explicit `hidden` property |
| `IN-007` | Operability | Low | Unbounded input — no constraints |
| `CP-001` | Operability | Info | No capabilities or outputs defined |

The engine also emits findings that are not tied to a single catalog rule ID, such as **Unused Input** (an input declared but never referenced via `get_input`), **Single Point of Failure**, and **Circular Dependency Detected**.

> The exact rule set evolves between releases. Treat `docs/RISK_RULES.md` as the source of truth — each finding's rule reference links straight to it.

## `.riskignore` — Suppressing Findings

Not every finding is actionable in every blueprint. Place a **`.riskignore`** file next to the blueprint's root YAML to suppress findings. Each non-comment line is one of three forms:

| Form | Example | Suppresses |
| ---- | ------- | ---------- |
| Rule ID | `IN-005` | Every finding from that rule |
| Rule + node | `ND-001:isolated_node` | That rule, but only on the named node |
| Exact title | `No Input Groups Defined` | Any finding whose title matches exactly |

Lines beginning with `#` are comments. For example:

```text
# Suppress all missing-hidden findings
IN-005
# Suppress the isolated-node finding only on this node
ND-001:isolated_node
# Suppress a specific finding by exact title
No Input Groups Defined
```

**Malformed lines and patterns that match no finding produce warnings** rather than being silently ignored — so a stale rule in `.riskignore` won't quietly stop matching. When you generate the visualizer, BPA reports what it suppressed and flags anything that didn't match:

```text
.riskignore line 8: "ZZ-999" did not match any finding — remove it or fix the pattern.
.riskignore suppressed 6 of 24 findings.
```

Suppressed findings are removed from the panel and counted in the metadata (`suppressedRiskCount`), so reviewers can see that suppressions are in effect.

## Component-Mode Risk Analysis (Composer Blueprints)

For **composer** blueprints — those that orchestrate `dell.nodes.ServiceComponent` sub-deployments — the analyzer runs in **component mode**. It recurses into each resolved component's sub-deployment topology and surfaces the referenced blueprint's findings alongside the composer's own.

- Findings that originate in a referenced blueprint are **prefixed with the blueprint ID**, e.g. `[network-base] Unpinned Plugin`, and the description notes the sub-deployment and node they came from (*"In sub-deployment 'network-base' (node 'vlan1'): …"*).
- Component mode adds composer-specific rules on top of the standard catalog:
  - **`BS-005` — Unresolved Component Definition** (High): a `ServiceComponent` whose referenced blueprint could not be resolved.
  - **Multi-Stage Orchestration Risk** (reliability): a failure in any stage can leave earlier stages deployed, requiring manual reconciliation.
  - **Component Sequencing Dependency** (operability): downstream components will not start until their dependencies complete successfully.
  - **Recursive Component Reference**: guards against a component that (directly or transitively) references itself.

### ServiceComponent Sub-Flow Panel

Selecting a component node that has a **resolved** sub-deployment opens the **Sub-Deployment** panel, which shows the referenced blueprint's nodes in **install order** — so you can drill from the composer into what each component actually deploys without leaving the visualizer.

## Example: A Multi-Tier Blueprint

Consider a blueprint that provisions a VM, attaches storage, and configures an application agent — a common three-layer pattern. After running BPA to generate the visualizer, the output would look something like this.

**Execution surface** — the graph shows infrastructure nodes (VM, datastore) upstream, with automation and storage nodes downstream in the order the install flow deploys them. External resources are marked distinctly.

**Step Details (install)** — steps derived from the dependency graph might look like:

| Step | Mode | What runs |
| ---- | ---- | --------- |
| 1 | Parallel | Independent infrastructure nodes (e.g. external datastore, cloud-init parser) |
| 2 | Parallel | Nodes that depend only on step 1 (e.g. cloud-init configs) |
| 3 | Sequential | Nodes with a single dependency chain (e.g. VM) |
| 4+ | Sequential | Application and data-layer nodes that depend on the VM |

**Input Groups** — input groups mirror the DAP deployment form. A three-layer blueprint might expose groups for compute configuration, networking, storage configuration, and application credentials.

**Risk Analysis** — the analyzer flags issues such as an unpinned plugin version (`IM-003`), an input missing a description (`IN-004`), or an external datastore that must pre-exist (`ND-002`), each with a severity, rule reference, and recommended fix.

**Plugins** — one entry per imported plugin, showing the plugin name and pinned version range.

> For a complete real-world example, see the `vm_parent-visualizer.html` file in the [VMaaS_Block_Storage_attach_PPDM](https://github.com/isg-tme-tech-ops/VMaaS_Block_Storage_attach_PPDM) repository.

## Technical Notes

- **No server required** — the HTML file is fully self-contained. Open with `File > Open` in any browser.
- **No CDN dependencies** — the visualizer inlines React (MIT license) directly into the HTML. It contains no `cytoscape`, `unpkg`, or `jsdelivr` references and works in air-gapped environments.
- **Data is embedded** — a `BLUEPRINT_DATA` constant holds all parsed blueprint information (metadata, nodes, relationships, input groups, input details, risks, plugins). The file does not re-read YAML at runtime.
- **Risk analysis is static and embedded** — findings are computed at generation time from the parsed blueprint and stored in `BLUEPRINT_DATA.risks`; the panel does not call out to any service at runtime. Node risk badges and Risk-panel node links cross-navigate to the same underlying nodes.
- **`.riskignore` is applied at generation time** — suppressed findings are removed from the data and counted in `BLUEPRINT_DATA.metadata.suppressedRiskCount`; unmatched patterns are reported as warnings.
- **Component-mode is automatic** — the analyzer switches to component mode when the blueprint contains resolvable `dell.nodes.ServiceComponent` nodes (`metadata.architectureMode: component`).
- **YAML syntax highlighting** is implemented without a library — a small regex-based tokenizer handles keys, strings, comments, booleans, numbers, and anchors.
