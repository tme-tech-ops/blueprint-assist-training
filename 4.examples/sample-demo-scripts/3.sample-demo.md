# Blueprint Assist Demo Script

- Audience: DevOps engineers, infrastructure teams
- Topics: Blueprint upload, deployment creation, install workflow, execution monitoring
- Key Skill: Taking a validated blueprint from local to a live DAP orchestrator deployment

## Session Setup

- **Session Type**: Fresh session
- **Bypass Mode**: Enabled

## Notes

- Requires a validated blueprint from Demo 2 before starting
- Replace `<execution_id>` with the ID returned by step 3.5
- Use `--trust-all` for dev orchestrators with self-signed certificates
- The Monitor Agent (step 3.13) automates the full lifecycle and is optional for time-constrained demos

## Demo Steps - Demo 3

### 3.1 Show target orchestrator connections

```text
Show me the target orchestrator connections that you have available

For self-signed certificates, connect with `--trust-all`
```

### 3.2 Show plugins available on the orchestrator

```bash
dap-bpa orchestrator plugins list --orchestrator stamp3 --trust-all
```

### 3.3 Upload the blueprint

```text
Upload this blueprint to the stamp3 orchestrator — do not deploy it yet:

Blueprint Path: 4.examples/target-build-folder/WindowsVM_Vsphere_demo
Blueprint ID: WindowsVM_Vsphere_demo
Revision: 1.0.0
```

### 3.4 Confirm upload status

```bash
dap-bpa orchestrator blueprints get WindowsVM_Vsphere_demo --fields id state --orchestrator stamp3 --trust-all
```

### 3.5 Create a deployment

The create deployment flow can be from the UI or CLI.

For this example, we will use the the AI tooling with the `dap-bpa` CLI to create the deployment

```text
Create a deployment from the uploaded blueprint using the example inputs:

Orchestrator: stamp3
Blueprint ID: WindowsVM_Vsphere_demo
Deployment ID: my-windows-vm-001
Inputs: 4.examples/target-build-folder/WindowsVM_Vsphere_demo/example_JSON_configs/demo3-json.json
```

### 3.6 Option to restart the installed workflow

```bash
dap-bpa orchestrator executions start --deployment-id my-windows-vm-001 --workflow-id install --orchestrator stamp3 --trust-all
```

### 3.7 Check execution status

Ask the AI tooling to poll until the deployment status shows as "completed" or "failed":

```text
Poll the execution status until it shows as "completed" or "failed":

dap-bpa orchestrator deployments get my-windows-vm-001 --fields id deployment_status installation_status latest_execution_id --orchestrator stamp3 --trust-all
```

Or, run the dap-bpa CLI command directly in a separate session to monitor the deployment:

```bash
dap-bpa orchestrator deployments get my-windows-vm-001 --fields id deployment_status installation_status latest_execution_id --orchestrator stamp3 --trust-all
```

### 3.8 Return the deployment capabilities

>Note the values are returned as their definitions (intrinsic functions), not resolved values, in this response. For authoritative resolved capabilities you currently
>need the DAP UI or the REST API endpoint directly:

```bash
dap-bpa orchestrator deployments get my-windows-vm-001 --fields id capabilities outputs --orchestrator stamp3 --trust-all
```

### 3.9 Stream live events

That command needs an execution ID, not the deployment ID. If using the AI tooling, it should get the execution ID from the deployment status response.

```bash
dap-bpa orchestrator events get my-windows-vm-001 --orchestrator stamp3 --trust-all
```

### 3.10 Uninstall the deployment

Uninstall the workflow. This will decommission the VM.

```bash
dap-bpa orchestrator executions start --deployment-id my-windows-vm-001 --workflow-id uninstall --orchestrator stamp3 --trust-all
```

### 3.11 Monitor the uninstall process

The same as section 3.7 - Ask the AI tooling to poll until the deployment status shows as "completed" or "failed". or run the CLI directly in a separate session.

```text
Poll the execution status until it shows as "uninstalled":

dap-bpa orchestrator deployments get my-windows-vm-001 --fields id deployment_status installation_status latest_execution_id --orchestrator stamp3 --trust-all
```

### 3.12 You can now delete the deployment

>The dap-bpa CLI does not support deleting deployments. The only supported deployment operations are list, get, create, and update.

To delete my-windows-vm-001, use one of these instead:

- DAP Orchestrator UI — open the deployment and use Delete (it's already uninstalled, so it can be safely removed).
- REST API — DELETE /api/v3.1/deployments/my-windows-vm-001 against the orchestrator.

This matches the documented CLI limitation: secrets delete, blueprints delete, and deployments delete are not available in the CLI — deletions go through the UI or API.

## Optional Demo Steps

### 3.13 Run the Monitor Agent (end-to-end automation)

Deploy straight from a local blueprint file and monitor end-to-end. Deploys the blueprint, then monitors until the deployment completes.

For reference, the `dap-bpa` on Windows the bpa-daemon.exe lands in C:\Users\<username>\bin.

Use the Monitor Agent to automate the full lifecycle — upload, deploy, install, and monitor:

> Note --file expects the blueprint YAML file, not the directory — it derives the blueprint ID from the filename (path.basename minus extension) --inputs expects the JSON file with the inputs for the blueprint

```bash
dap-bpa monitor --file 4.examples/target-build-folder/WindowsVM_Vsphere_demo/blueprint.yaml --inputs "4.examples/target-build-folder/WindowsVM_Vsphere_demo/example_JSON_configs/demo3-json.json" --orchestrator stamp3 --trust-all
```

### 3.14 Check Monitor Agent as Detached (background) mode + status checks

```bash
# start in background
dap-bpa monitor --deployment-id my-windows-vm-001 --detach \
  --orchestrator stamp3 --trust-all

# check the most recent session
dap-bpa monitor --status --orchestrator stamp3 --trust-all

# or a specific session/execution
dap-bpa monitor --status --execution-id <exec-id> --orchestrator stamp3 
--trust-all

# is the background daemon alive?
dap-bpa monitor --daemon-status

# stop the background daemon when done
dap-bpa monitor --daemon-stop
```
