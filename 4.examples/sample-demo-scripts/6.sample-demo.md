# Blueprint Assist Demo Script

- Audience: DevOps engineers, security teams, blueprint authors
- Topics: Secret types, creation, `get_secret` function, best practices
- Key Skill: Secure credential management in blueprints

## Session Setup

- **Session Type**: Fresh session
- **Bypass Mode**: Enabled

## Notes

- Orchestrator steps require DAP credentials — use `--trust-all` for dev/self-signed certs
- Replace `<orchestrator>` in any command with a profile name returned by `dap-bpa status`
- Never display real credential values on screen; use placeholder values for the recording
- Uses the blueprint from Demo 2 at `4.examples/target-build-folder/WindowsVM_Vsphere_demo`
- The CLI does not support deleting secrets — deletions go through the DAP UI or REST API

## Demo Steps - Demo 6

### 6.1 Show the available secret types

```bash
dap-bpa knowledge secret-type list
```

### 6.2 Inspect a specific secret type schema

```text
Explain the schema for a specific secret type and what fields it expects
```

```bash
dap-bpa knowledge secret-type get <secret-type>
```

### 6.3 List existing secrets on the orchestrator

```bash
dap-bpa orchestrator secrets list --orchestrator <orchestrator> --trust-all
```

### 6.4 Create a secret

Create a secret once so it can be referenced by many blueprints. Use a placeholder value for the demo.

```bash
dap-bpa orchestrator secrets create --key demo_vsphere_password --value "REPLACE_ME" --display-name "Demo vSphere Password" --description "vSphere credential for demo VM" --orchestrator <orchestrator> --trust-all
```

### 6.5 Inspect the created secret

```bash
dap-bpa orchestrator secrets get demo_vsphere_password --orchestrator <orchestrator> --trust-all
```

### 6.6 Show how the blueprint references the secret

```text
Show me how this blueprint references secrets. Explain how get_secret combines with get_input to inject credentials securely, using the blueprint at 4.examples/target-build-folder/WindowsVM_Vsphere_demo
```

### 6.7 Review secret handling best practices

```text
Review this blueprint's credential handling and confirm it follows best practices — no hardcoded passwords, secret_key inputs with no defaults, and get_secret used for every credential
```

## Notes

- Secret types define credential schemas
- Create secrets once, reference them many times across blueprints
- Use `get_secret` with `get_input` for secure credential injection
- Never hardcode passwords, API keys, or certificates in a blueprint
- `secret_key` inputs must have NO default value
